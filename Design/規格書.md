## 1. 文件目的

本規格書整理簡易 BPM 系統的功能需求、業務規則與限制，作為開發、測試與驗收的共同依據。內容來自需求討論與問答所確認之決策。

---

## 2. 專案背景

### 2.1 目標

建置一套供**內部單位**使用的簡易 BPM（Business Process Management）系統，支援：

- 帳號與權限管理
- 電子表單填寫與送出
- 可設定的多關簽核流程（含會簽）
- 收件夾與待辦管理
- 依表單資料與簽核進度，自動產生紙本格式文件（PDF），並在指定位置疊加簽核者個人電子章

### 2.2 使用對象與規模

| 項目 | 說明 |
|------|------|
| 主要使用者 | 內部同仁 |
| 預估人數 | 約 10–30 人 |
| 使用情境 | 內部行政流程（第一版以經費請款為主） |
| 對外開放 | 第一版**不**提供公開申請或自助註冊 |

### 2.3 部署策略

- **第一階段**：本機開發與原型驗證
- **上線環境**：尚未決定（雲端 / 內網待後續規劃）

## 3. 範圍定義

### 3.1 第一版包含（In Scope）

| 模組 | 說明 |
|------|------|
| 帳號管理 | Email + 密碼登入；管理員建立/停用帳號 |
| 個人電子章 | 每位簽核者上傳並維護自己的印章圖檔 |
| 表單 | 經費/請款申請（固定欄位）；架構可擴充第二種表單 |
| 流程設定 | 每種表單可設定不同簽核流程（順序關、會簽關） |
| 收件夾 | 待簽、我送出、已完成、代理相關檢視 |
| 簽核操作 | 同意、退回重填、簽核意見、代理簽核、催辦 |
| 文件產生 | 上傳 PDF 模板；欄位與印章依座標對應；**每關簽核後更新 PDF** |
| 通知 | Email；Discord Bot **私訊**待簽者 |
| 稽核 | 簽核歷程與意見紀錄 |

### 3.2 第一版不包含（Out of Scope）

| 項目 | 備註 |
|------|------|
| 自助註冊 | 帳號僅由管理員建立 |
| 視覺化表單設計器 | 預留架構，第二版再實作 |
| 條件式流程分支 | 如「金額超過 X 走不同路徑」 |
| 欄位級變更稽核 | 僅保留簽核歷程與意見 |
| PDF 數位簽章 | 以圖檔疊加印章，非 CA 憑證簽章 |
| Word 模板 / 郵件合併 | 統一以 PDF 為底稿 |
| LINE 等即時通訊 | 可於後續版本評估 |

### 3.3 後續版本預留

- 新增其他表單類型（請假單、用印申請等）
- 拖拉式表單設計器
- 條件式路由、欄位級稽核

## 4. 功能需求

### 4.1 帳號與驗證（FR-AUTH）

| 編號 | 需求 | 優先級 |
|------|------|--------|
| FR-AUTH-01 | 使用者以 **Email + 密碼** 登入 | 必須 |
| FR-AUTH-02 | **無自助註冊**；僅管理員可建立、停用帳號 | 必須 |
| FR-AUTH-03 | 管理員可重設使用者密碼 | 必須 |
| FR-AUTH-04 | 角色至少區分：`admin`（系統管理）、`user`（一般使用者） | 必須 |
| FR-AUTH-05 | Session 登入狀態維持；未登入無法存取業務頁面 | 必須 |

### 4.2 個人設定與電子章（FR-SEAL）

| 編號 | 需求 | 優先級 |
|------|------|--------|
| FR-SEAL-01 | 簽核者可上傳**個人電子章**圖檔（建議 PNG 透明底） | 必須 |
| FR-SEAL-02 | 簽核者可更換、刪除自己的印章 | 必須 |
| FR-SEAL-03 | 簽核同意時，系統使用**該簽核者**的印章疊加至文件指定位置 | 必須 |
| FR-SEAL-04 | 未上傳印章時，簽核應提示補齊或阻擋（實作時擇一並於 UI 說明） | 應有 |

### 4.3 表單（FR-FORM）

| 編號 | 需求 | 優先級 |
|------|------|--------|
| FR-FORM-01 | 第一版提供**經費/請款申請**表單 | 必須 |
| FR-FORM-02 | 申請人可儲存草稿、編輯後送出 | 必須 |
| FR-FORM-03 | 送出後建立流程實例並進入簽核 | 必須 |
| FR-FORM-04 | 表單欄位定義採固定結構（程式/設定），非拖拉設計器 | 必須 |
| FR-FORM-05 | 架構上可新增第二種 `FormTemplate`，無需重寫核心 | 應有 |

#### 4.3.1 經費請款表單欄位（MVP）

| 欄位 | 類型 | 說明 |
|------|------|------|
| 申請日期 | 日期 | 可預設為當日 |
| 請款人 | 文字（唯讀） | 自動帶入登入使用者姓名 |
| 用途說明 | 多行文字 | 必填 |
| 費用明細 | 明細表 | 可新增多列：日期、項目、金額、備註 |
| 合計金額 | 數字（唯讀） | 依明細自動加總 |
| 付款方式 / 受款資訊 | 文字 | 受款人、帳號等 |
| 附件 | 檔案（選填） | 收據 PDF 或圖片 |

#### 4.3.2 簽核流程範例（可於後台修改）

預設示範：`申請人 → 單位主管 → 會計 → 理事長`  
亦可設定會簽關（例如財務與秘書同時簽）。**實際關卡名稱與人數待單位確認。**

### 4.4 流程與簽核（FR-WF）

| 編號 | 需求 | 優先級 |
|------|------|--------|
| FR-WF-01 | **每種表單**可綁定獨立的簽核流程範本 | 必須 |
| FR-WF-02 | 流程步驟支援**依序**（serial）與**會簽**（parallel，全部同意才前進） | 必須 |
| FR-WF-03 | 步驟可指派：指定使用者、依角色、或由表單欄位決定簽核人 | 必須 |
| FR-WF-04 | 簽核者可以**同意**或**退回**；退回時申請人可修改後重送 | 必須 |
| FR-WF-05 | 退回時**必須填寫**簽核意見 | 必須 |
| FR-WF-06 | 同意時可填寫意見（選填或必填，實作時於規格附錄統一） | 應有 |
| FR-WF-07 | 支援**代理簽核**：使用者設定代理人與有效期，待辦可改由代理人處理 | 必須 |
| FR-WF-08 | 流程結案後狀態為已完成，不可再編輯表單內容（除非另開新單） | 必須 |

### 4.5 收件夾（FR-INBOX）

| 編號 | 需求 | 優先級 |
|------|------|--------|
| FR-INBOX-01 | **待我簽核**：目前使用者（或代理身份）待處理的 Task | 必須 |
| FR-INBOX-02 | **我送出的**：自己申請的流程與狀態 | 必須 |
| FR-INBOX-03 | **已完成**：已結案案件查詢 | 必須 |
| FR-INBOX-04 | **我代理的**：代理他人簽核的相關待辦/紀錄 | 必須 |
| FR-INBOX-05 | 簽核頁顯示：表單內容、簽核歷程、目前 PDF 預覽 | 必須 |
| FR-INBOX-06 | **催辦**：對逾期待簽案件發送催辦通知；同一案件 24 小時內不重複催辦 | 必須 |

### 4.6 文件產生（FR-DOC）

| 編號 | 需求 | 優先級 |
|------|------|--------|
| FR-DOC-01 | 管理員可上傳**空白 PDF** 作為表單紙本模板 | 必須 |
| FR-DOC-02 | 後台提供**座標對應**：將表單欄位對應至 PDF 上 `(page, x, y)` 與字級 | 必須 |
| FR-DOC-03 | 後台可為**每個簽核步驟**設定印章放置區（矩形：頁碼、座標、寬高） | 必須 |
| FR-DOC-04 | 申請人送出後產生 **v1 PDF**（填入表單欄位，尚未蓋章或僅蓋已完成的章） | 必須 |
| FR-DOC-05 | **每一關簽核同意後**重新產生 PDF 新版本，疊加該關及之前已完成關卡之印章 | 必須 |
| FR-DOC-06 | 使用者可預覽、下載目前版本的 PDF | 必須 |
| FR-DOC-07 | 後台提供「測試產生 PDF」以校正座標 | 應有 |
| FR-DOC-08 | 產出格式以 **PDF** 為主；JPG 匯出列為後續評估 | 可選 |

#### 4.6.1 PDF 合成業務規則

1. 底稿：`DocumentTemplate` 上傳的原始 PDF。
2. 文字：依 `fieldMappings` 將 `FormData` 繪製至對應位置。
3. 印章：僅對**已完成簽核**的步驟，在對應 `sealSlot` 嵌入**實際簽核者**的個人電子章，等比縮放至區塊內。
4. 版本：每次簽核同意後遞增 `DocumentVersion`，保留檔案路徑供下載與稽核參照。

### 4.7 通知（FR-NOTIFY）

| 編號 | 需求 | 優先級 |
|------|------|--------|
| FR-NOTIFY-01 | 新任務指派時發送 **Email** 通知簽核者 | 必須 |
| FR-NOTIFY-02 | 退回、催辦時發送 Email | 必須 |
| FR-NOTIFY-03 | Email 內含深連結，可直接開啟對應待辦 | 應有 |
| FR-NOTIFY-04 | 新任務指派時以 **Discord Bot 私訊** 通知簽核者 | 必須 |
| FR-NOTIFY-05 | 使用者須於個人設定完成 **Discord OAuth2 綁定** 後才可收到私訊 | 必須 |
| FR-NOTIFY-06 | Discord 發送失敗時記錄 log，並以 Email 為主要送達管道 | 應有 |

### 4.8 管理後台（FR-ADMIN）

| 編號 | 需求 | 優先級 |
|------|------|--------|
| FR-ADMIN-01 | 使用者管理（建立、停用、重設密碼） | 必須 |
| FR-ADMIN-02 | 表單範本與流程範本設定 | 必須 |
| FR-ADMIN-03 | PDF 模板上傳與欄位/印章座標編輯 | 必須 |

### 4.9 稽核（FR-AUDIT）

| 編號 | 需求 | 優先級 |
|------|------|--------|
| FR-AUDIT-01 | 記錄每筆簽核：操作者、動作（同意/退回/代理）、時間、意見 | 必須 |
| FR-AUDIT-02 | 申請人與簽核者可檢視完整簽核歷程 | 必須 |
| FR-AUDIT-03 | 第一版**不**記錄表單欄位逐項變更 diff | 不在範圍 |

## 5. 非功能需求

| 編號 | 類別 | 需求 |
|------|------|------|
| NFR-01 | 語系 | 使用者介面為**繁體中文** |
| NFR-02 | 瀏覽器 | 支援現代桌面瀏覽器（Chrome、Edge、Firefox 近期版本） |
| NFR-03 | 安全性 | 密碼雜湊儲存；HTTPS 於正式環境啟用 |
| NFR-04 | 效能 | 10–30 人同時使用，單次 PDF 產生於數秒內完成（一般 A4 單頁至數頁） |
| NFR-05 | 可維護性 | 本機以 SQLite 開發；上線可切換 PostgreSQL，僅變更連線設定 |
| NFR-06 | 檔案 | 模板、印章、產出 PDF 存於 `storage/`（開發環境）；正式環境可改物件儲存 |

## 8. 需求決策紀錄（問答摘要）

| 主題 | 決策 |
|------|------|
| 使用規模 | 內部約 10–30 人 |
| 部署 | 先本機原型 |
| 技術棧 | 由開發方建議（見實作計劃：Next.js + Prisma + pdf-lib 等） |
| 表單模型 | 先固定 2–3 種能力中的**經費單**；之後擴充設計器 |
| 流程複雜度 | 每種表單可設定流程（含會簽） |
| 文件模板 | 上傳 PDF + 後台座標對應 |
| 印章 | 每位簽核者個人章 |
| 登入 | Email + 密碼，管理員建帳 |
| 文件更新時機 | 每個簽核關卡更新 PDF |
| 收件夾 | 完整功能（含代理、退回意見、催辦） |
| 通知 | Email + Discord Bot 私訊 |
| Discord 實作 | Bot 私訊（非僅 Webhook 頻道） |
| 稽核深度 | 僅簽核歷程與意見 |
| MVP 表單 | 經費/請款申請 |

## 9. 驗收準則（摘要）

1. 管理員可建立帳號，使用者可以 Email 登入。
2. 簽核者可上傳個人章；未上傳時行為符合 UI 說明。
3. 申請人可完成經費請款填寫、送出，並於「我送出的」追蹤狀態。
4. 依設定流程派送待辦；會簽關需全部同意才進下一關。
5. 簽核者可同意/退回；退回後申請人可修改重送。
6. 代理簽核與催辦功能可用，且催辦有 24 小時節流。
7. 每關同意後可下載含正確欄位與累積印章的 PDF。
8. 待簽時收到 Email；已綁定 Discord 者收到 Bot 私訊。
9. 簽核歷程完整可查。

## 11. 詞彙表

| 詞彙 | 說明 |
|------|------|
| BPM | Business Process Management，商業流程管理 |
| 流程實例（Instance） | 一筆已送出的表單及其簽核狀態 |
| Task | 指派給某位簽核者的待辦項目 |
| 會簽（parallel） | 同一關多位簽核者皆須同意 |
| 印章槽（seal slot） | PDF 上預留給該簽核步驟蓋章的矩形區域 |
| DocumentVersion | 某次簽核後產出的 PDF 檔案版本 |
